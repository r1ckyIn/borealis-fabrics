**PurchaseOrderImportStrategy** (`purchase-order-import.strategy.ts`):

This strategy handles 海宁优途-采购单 format. Per CONTEXT.md locked decision, it creates BOTH products/prices AND purchase order records (Order with OrderItems).

**Product + Price creation** (same as before):
- Extracts product/price data from purchase order items
- Creates Product records (with subCategory inferred from item names) and ProductSupplier pricing

**Order creation** (per CONTEXT.md locked decision):
- Creates an Order record for each purchase order file imported
- The Order model requires `customerId`. Since purchase orders are TO suppliers (not FROM customers), use a "self-customer" pattern: look up or create a Customer record for "铂润" (Borealis Fabrics, the company itself) during `getExistingKeys()`. This represents the company placing orders to suppliers.
- Set Order fields: `customerId` = the "铂润" self-customer id, `status` = 'INQUIRY', `notes` = '采购单 (Purchase Order) - 海宁优途 - [PO date from filename]'
- Create OrderItem records for each line item: set `productId` from newly created/looked-up Product, `supplierId` from supplier lookup (海宁优途), `quantity`, `unit`, `salePrice` = 0 (not applicable for purchase orders), `purchasePrice` = unit price from PO, `subtotal` = quantity * unitPrice, `deliveryDate` from PO line item
- The `createBatch()` method should: (1) create/update Products + ProductSupplier pricing in transaction, (2) create Order + OrderItems in same transaction

Key implementation details:
- **Non-standard layout**: Data starts at row 5 (rows 1-3 are metadata, row 4 is headers)
- **Header detection**: `matchesHeaders()` checks for presence of columns like '订单', 'PO#', '名称', '数量', '单价' (use case-insensitive substring matching, NOT exact match)
- **Column mapping** (1-based from row 4 headers):
  - Col 1: 订单PO# (purchase order number -- stored in Order notes for reference)
  - Col 2: 名称 (product name)
  - Col 3: 规格型号 (specification/model)
  - Col 4: 单位 (unit)
  - Col 5: 数量 (quantity)
  - Col 6: 单价 (unit price = purchasePrice)
  - Col 7: 交货日期 (delivery date)
  - Col 8: 备注 (notes)
- **SubCategory inference**: If name contains '铁架' or '架' -> IRON_FRAME. If name contains '电机' -> MOTOR. If name contains '床垫' -> MATTRESS. Otherwise -> ACCESSORY.
- **Supplier**: The supplier is "海宁优途" (hardcoded from file name context). Look up in supplier map during getExistingKeys(). If not found, all rows fail with "Supplier '海宁优途' not found".
- **Self-customer lookup**: During getExistingKeys(), look up a customer with companyName containing '铂润'. If not found, create one with companyName='铂润面料' (the company itself) and minimal fields.
- **Dedup key**: `specification + '::' + name` (specification acts like modelNumber for PO items)
- **createBatch**: Create Product + ProductSupplier + Order + OrderItems in transaction. Generate productCode via CodeGeneratorService and orderCode via CodeGeneratorService.

The constructor injects PrismaService, CodeGeneratorService, and RedisService (for code generation).

**SalesContractImportStrategy** (`sales-contract-import.strategy.ts`):

This strategy handles BOTH 购销合同 (2 files) AND 客户订单 (6 files) -- total 8 files. Per RESEARCH.md analysis, the 客户订单 files (77947/77955/77962 series) use the EXACT same layout template as 购销合同. There is NO separate CustomerOrderImportStrategy; this single strategy handles all 8 files.

It creates Order records with OrderItem records.

Key implementation details:
- **Non-standard layout**: Metadata in rows 1-8, column headers at row 9, data from row 10+
- **Metadata extraction** from rows 1-8: Extract customer name, contract number, date from fixed positions. The customer name is typically in row 2-3 area. Contract number in row 1.
- **Two column variants** handled by the same strategy:
  - Fabric variant (面料): columns = 面料名称 | 涂层 | 系列 | 色号 | 颜色中文 | 颜色英文 | 单位 | 数量 | 单价 | 金额 | 交货日期 | PI.# | 生产单号
  - Iron frame variant (铁架): columns = 品名 | 规格 | 单位 | 数量 | 单价 | 金额 | 交货日期 | PI.# | 生产单号 | 型号 | 备注
- **Variant detection**: Scan header row (row 9) for '面料名称' (fabric) vs '品名' (product/iron frame).
- **Header detection**: `matchesHeaders()` checks for '面料名称' or '品名' combined with '数量' and '单价' or '金额'. NOTE: Headers may be RichText -- use getCellValue to extract text.
- **Entity lookups in getExistingKeys()**:
  - Pre-load customer name-to-id map
  - Pre-load fabric name-to-id map (for fabric variant)
  - Pre-load product name-to-id map (for iron frame variant)
  - Pre-load existing order codes for dedup
- **validateRow** checks:
  - Item name exists (面料名称 or 品名)
  - Quantity > 0
  - Unit price > 0
  - Referenced fabric/product found in system (error if not: "Fabric 'xxx' not found in system, import base data first")
- **transformRow** extracts: itemName, quantity, unitPrice, unit, subtotal, deliveryDate, notes, piNumber, productionNumber, and flags whether fabric or product variant
- **createBatch**: Creates one Order per file (or per contract number), with multiple OrderItems. Uses CodeGeneratorService to generate orderCode. Sets status = 'INQUIRY', customerId from metadata.
  - For fabric items: set fabricId from fabric lookup
  - For product items: set productId from product lookup
  - Calculate totalAmount as sum of item subtotals

Inject PrismaService, CodeGeneratorService, RedisService (for order code generation).

**Both strategies** use getCellValue and parseNumber from `../utils/excel.utils.ts`.

**Important**: These strategies are NOT auto-detected by `detectStrategy()`. They have dedicated import endpoints because their file layouts are fundamentally different from template files. The `importData()` private method assumes row-1 headers and flat tabular data -- these strategies need custom orchestration methods in ImportService.

```typescript
private async importNonStandardData(
  file: Express.Multer.File,
  strategy: ImportStrategy,
  headerRowNumber: number,
  dryRun = false,
): Promise<ImportResultDto> {
  const workbook = new ExcelJS.Workbook();
  try {
    await workbook.xlsx.load(file.buffer as unknown as ArrayBuffer);
  } catch {
    throw new BadRequestException('Invalid Excel file format');
  }

  const worksheet = workbook.worksheets[0];
  if (!worksheet || worksheet.rowCount < headerRowNumber + 1) {
    throw new BadRequestException('Excel file has insufficient rows');
  }

  const existingKeys = await strategy.getExistingKeys();
  const batchKeys = new Set<string>();
  const failures: ImportFailureDto[] = [];
  let skippedCount = 0;
  const entitiesToCreate: Record<string, unknown>[] = [];

  worksheet.eachRow((row, rowNumber) => {
    if (rowNumber <= headerRowNumber) return; // Skip metadata + header

    const result = strategy.validateRow(row, rowNumber, batchKeys, existingKeys);
    if (result.skipped) { skippedCount++; return; }
    if (!result.valid) { if (result.failure) failures.push(result.failure); return; }

    const key = strategy.getRowKey(row);
    batchKeys.add(key);
    entitiesToCreate.push(strategy.transformRow(row));
  });

  if (entitiesToCreate.length > 0 && !dryRun) {
    await strategy.createBatch(entitiesToCreate);
  }

  return { successCount: entitiesToCreate.length, skippedCount, failureCount: failures.length, failures };
}
```
