# Phase 04.1: 付款凭证强制上传 - Context

**Gathered:** 2026-03-24
**Status:** Ready for planning

<domain>
## Phase Boundary

All payment status changes (customer and supplier) must include uploaded voucher files (images/PDF/Office). Supports multi-file upload, drag-and-drop, preview/download, and append-only modification. This phase covers backend schema changes, API enforcement, and frontend upload integration into existing payment flows.

</domain>

<decisions>
## Implementation Decisions

### Mandatory Voucher Rules
- ALL payment status changes require at least one voucher — both `unpaid→partial`, `partial→paid`, and any other status transitions
- Applies to BOTH customer payments and supplier payments equally
- Supported file types: JPG/PNG/WEBP images, PDF, Word/Excel (Office documents)
- No limit on number of voucher files per payment operation (single file size limit applies)
- Each voucher can have an optional text remark/note
- Double enforcement: frontend disables save button when no vouchers attached + backend returns 400 if no vouchers in request

### Voucher Modification Policy
- Append-only: uploaded vouchers CANNOT be deleted or replaced
- Users can only add new vouchers to existing payment records
- This ensures audit trail integrity — every payment operation's evidence is preserved

### Voucher Display
- Vouchers displayed as file list (filename + upload time + size + action buttons)
- Customer and supplier vouchers displayed separately in their respective tabs (matching existing PaymentStatusCard tab structure)
- Image files: click to preview in modal (reuse existing preview pattern)
- PDF files: click opens in new browser tab
- Office files: click to download

### Upload Interaction
- Drag-and-drop + click upload (Ant Design Upload.Dragger)
- Upload area appears in payment edit modal, below payment form fields
- Files uploaded immediately on selection; save button submits payment data + voucher associations atomically

### Claude's Discretion
- Voucher association model: bind to PaymentRecord (per-operation tracking) vs Order-level vs new PaymentEvent table — choose best practice for audit trail + simplicity balance
- Payment operation flow adjustment: whether to add upload area in existing Modal or restructure the flow
- File size limit per file (recommend 10MB)
- Upload progress indicator design
- Database schema for voucher-payment association (new table vs extending PaymentRecord)
- Backend validation error messages for missing vouchers

</decisions>

## Existing Code Insights

### Reusable Assets
- `FileService` + `StorageProvider`: fully functional file upload with local/COS dual-mode — reuse directly for voucher uploads
- `file.api.ts`: `uploadFile()` and `uploadFiles()` already handle FormData + progress — reuse for voucher upload
- `ImageUploader` component: has drag-drop, preview, multi-file — can be adapted/extended for mixed file types (currently image-only)
- `PaymentStatusCard`: displays payment info with edit button — voucher count/indicator could be added here
- `PaymentRecord` model: already has orderId, type (customer/supplier), supplierId, amount, payMethod, remark — potential target for voucher association
- `OrderPaymentSection`: has customer/supplier payment tabs with edit modals — voucher upload area integrates here

### Established Patterns
- File storage: database stores key-only, URL generated at read-time via StorageProvider.getUrl()
- API client unwraps ApiResponse<T> — consumers receive data directly
- Error handling: ERROR_CODE_MESSAGES > HTTP_STATUS > raw message > fallback
- Form pages: Form.useForm() at page level, passed to sub-components
- TanStack Query hooks for data fetching/mutation

### Integration Points
- `OrderPaymentSection` customer/supplier edit modals — add voucher upload area
- `order-payment.service.ts` updateCustomerPayment/updateSupplierPayment — add voucher validation
- `PaymentRecord` model or new table — store voucher file references
- `FileService` — reuse for voucher file storage
- Backend DTOs — add voucher file IDs to payment update requests

</code_context>

<specifics>
## Specific Ideas

- User explicitly chose "append-only" for vouchers — audit trail is important, no deletion/replacement allowed
- User wants all payment status changes enforced equally (not just paid/partial) — strictest mode
- File list display preferred over thumbnail grid — cleaner for mixed file types (images + PDF + Office)
- Drag-and-drop upload is required (Ant Design Upload.Dragger)
- Each voucher file can have an optional remark/note field

</specifics>
