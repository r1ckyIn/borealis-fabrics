    **Fix 1: Dev login bypass (auth.controller.ts lines 124-129)**

    Change the devLogin guard logic from:
    ```typescript
    const nodeEnv = this.configService.get<string>('nodeEnv');
    if (nodeEnv !== 'development') {
      throw new ForbiddenException('Dev login is only available in development mode');
    }
    ```
    To:
    ```typescript
    const nodeEnv = this.configService.get<string>('nodeEnv');
    const allowDevLogin = process.env.ALLOW_DEV_LOGIN === 'true';
    if (nodeEnv !== 'development' && !allowDevLogin) {
      throw new ForbiddenException(
        'Dev login is only available in development mode or when ALLOW_DEV_LOGIN=true',
      );
    }
    ```

    This implements RESEARCH recommendation Option A: explicit opt-in via env var, easy to remove for Phase B.

    **Fix 2: Cookie secure flag (auth.controller.ts lines 54-57)**

    Change getCookieOptions from:
    ```typescript
    private getCookieOptions(): typeof AUTH_COOKIE_OPTIONS & { secure: boolean } {
      const isProduction = this.configService.get<string>('nodeEnv') === 'production';
      return { ...AUTH_COOKIE_OPTIONS, secure: isProduction };
    }
    ```
    To:
    ```typescript
    private getCookieOptions(): typeof AUTH_COOKIE_OPTIONS & { secure: boolean } {
      const forceHttps = process.env.FORCE_HTTPS_COOKIES === 'true';
      return { ...AUTH_COOKIE_OPTIONS, secure: forceHttps };
    }
    ```

    Rationale: `secure: true` means browsers reject the cookie over HTTP. In Phase A (HTTP + IP), this blocks login entirely (Pitfall 4). Instead of tying `secure` to NODE_ENV, tie it to an explicit `FORCE_HTTPS_COOKIES` env var. Set it to `true` in Phase B when HTTPS is enabled.

    **Fix 3: Production config validation (configuration.ts)**

    Move WEWORK_* vars to a separate conditional block:
    ```typescript
    const requiredVars = [
      'JWT_SECRET', 'DATABASE_URL', 'REDIS_URL',
      'COS_SECRET_ID', 'COS_SECRET_KEY', 'COS_BUCKET', 'COS_REGION',
      'CORS_ORIGINS',
    ];

    // WeChat Work vars only required when dev login is NOT enabled
    if (process.env.ALLOW_DEV_LOGIN !== 'true') {
      requiredVars.push(
        'WEWORK_CORP_ID', 'WEWORK_SECRET', 'WEWORK_AGENT_ID', 'WEWORK_REDIRECT_URI',
      );
    }
    ```

    This allows Phase A to set dummy values for WEWORK_* (or omit them entirely when ALLOW_DEV_LOGIN=true), avoiding Pitfall 3 (container crashes on startup).

    **Create `docker-compose.prod.yml`** at project root:

    ```yaml
    # Production Docker Compose -- Borealis Fabrics
    # Usage: docker compose -f docker-compose.prod.yml up -d
    # MySQL: External CDB (not in compose). Set DATABASE_URL in backend/.env
    # Monitoring: Separate file (docker-compose.monitoring.yml)

    services:
      nestjs:
        build:
          context: ./backend
          dockerfile: Dockerfile
        restart: unless-stopped
        env_file: ./backend/.env
        depends_on:
          redis:
            condition: service_healthy
        healthcheck:
          test: ["CMD", "node", "-e", "fetch('http://localhost:3000/health').then(r=>{if(!r.ok)throw 1})"]
          interval: 30s
          timeout: 5s
          retries: 3
          start_period: 30s
        deploy:
          resources:
            limits:
              memory: 768M

      redis:
        image: redis:7-alpine
        restart: unless-stopped
        command: redis-server --maxmemory 128mb --maxmemory-policy allkeys-lru
        healthcheck:
          test: ["CMD", "redis-cli", "ping"]
          interval: 10s
          timeout: 5s
          retries: 5
        volumes:
          - redis_data:/data
        deploy:
          resources:
            limits:
              memory: 192M

      nginx:
        image: nginx:1.27-alpine
        ports:
          - "80:80"
        volumes:
          - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
          - ./nginx/conf.d:/etc/nginx/conf.d:ro
          - ./frontend/dist:/usr/share/nginx/html:ro
        depends_on:
          nestjs:
            condition: service_healthy
        restart: unless-stopped
        deploy:
          resources:
            limits:
              memory: 64M

    volumes:
      redis_data:
    ```

    Key decisions (per D-01, D-09, RESEARCH Pattern 3+4):
    - MySQL NOT in compose (CDB is external, per D-03)
    - NestJS: 768MB memory limit, healthcheck on /health, start_period: 30s for Prisma warmup
    - Redis: 128MB data + 64MB overhead = 192MB limit, allkeys-lru eviction
    - Nginx: 64MB, mounts frontend/dist (pre-built locally or in CI), mounts nginx config
    - Port 80 only (no 443 in Phase A per D-09)
    - Monitoring is in separate `docker-compose.monitoring.yml` (already exists)
    - `frontend/dist` mounted directly (built locally before deploy, or built in CI)
    - redis_data volume for persistence across restarts
