# Phase 04.1: 付款凭证强制上传 - Research

**Researched:** 2026-03-24
**Domain:** File upload + payment workflow integration (NestJS backend + React/Ant Design frontend)
**Confidence:** HIGH

## Summary

Phase 04.1 requires mandatory voucher file uploads for ALL payment status changes (customer and supplier). The existing codebase has strong foundations: `FileService` with `StorageProvider` abstraction (local/COS dual-mode), `file.api.ts` with `uploadFile()`/`uploadFiles()` + progress support, and an `ImageUploader` component with drag-drop. The existing `PaymentRecord` model in the schema is already designed for per-operation payment tracking but is **currently unused** (no `create` calls in backend services). This makes it the natural candidate for voucher association.

The core technical challenge is integrating voucher upload into the existing payment modal flow atomically: files upload immediately on selection (async), but the payment data + voucher associations must be submitted together on save. The frontend needs a new `VoucherUploader` component (extending beyond image-only to support PDF/Word/Excel), and the backend needs schema changes (new `PaymentVoucher` join table) plus DTO/service validation to enforce mandatory vouchers.

**Primary recommendation:** Create a `PaymentVoucher` table linking `PaymentRecord` to `File`, activate the dormant `PaymentRecord` model to track each payment operation with its vouchers, and build a `VoucherUploader` component using Ant Design `Upload.Dragger` with file-type-aware preview behavior.

## Architecture Patterns

### Recommended Schema Design

**Decision: Use `PaymentRecord` + new `PaymentVoucher` join table**

The `PaymentRecord` model already exists in `schema.prisma` (line 367) but is completely unused (no `.create()` calls in backend). This model was designed for per-operation payment tracking with `orderId`, `type` (customer/supplier), `supplierId`, `amount`, `payMethod`, `remark`, `operatorId`. Activating it provides:

1. **Audit trail per payment operation** -- each payment update creates a PaymentRecord
2. **Voucher-to-operation binding** -- vouchers attach to a specific payment operation, not just the order
3. **Append-only by design** -- new operations create new records, old records are immutable

```prisma
// New join table for voucher-payment association
model PaymentVoucher {
  id              Int      @id @default(autoincrement())
  paymentRecordId Int      @map("payment_record_id")
  fileId          Int      @map("file_id")
  remark          String?  @db.Text
  createdAt       DateTime @default(now()) @map("created_at")

  paymentRecord PaymentRecord @relation(fields: [paymentRecordId], references: [id], onDelete: Cascade)
  file          File          @relation(fields: [fileId], references: [id])

  @@index([paymentRecordId])
  @@index([fileId])
  @@map("payment_vouchers")
}

// Update PaymentRecord to include vouchers relation
model PaymentRecord {
  // ... existing fields ...
  vouchers PaymentVoucher[]
}

// Update File to include voucher relation
model File {
  // ... existing fields ...
  paymentVouchers PaymentVoucher[]
}
```

**Why this approach over alternatives:**
- **vs. Order-level binding**: Loses per-operation granularity. Cannot tell which voucher belongs to which payment event.
- **vs. JSON array on PaymentRecord**: No referential integrity, cannot query files independently, harder to enforce append-only.
- **vs. New PaymentEvent table**: PaymentRecord already exists with the right shape. Creating another table is redundant.

### Payment Flow (Updated)

```
Current flow:
  Frontend Modal → form.validateFields() → mutation.mutateAsync(paymentData) → backend updates Order/SupplierPayment

New flow:
  Frontend Modal → user uploads files (immediate upload to /files endpoint) → collects fileIds
  → form.validateFields() → mutation.mutateAsync({ paymentData, voucherFileIds, voucherRemarks })

```

### Anti-Patterns to Avoid
- **Uploading files inside the payment mutation**: Files should upload independently via existing `/files` endpoint. The payment mutation only receives `fileId[]` references. This keeps the file upload decoupled and allows progress indicators.
- **Storing file URLs in PaymentRecord directly**: Follow existing pattern -- File table stores key-only, URL resolved at read-time via StorageProvider.
- **Making vouchers deletable from frontend**: User decision is append-only. Do not add delete buttons on voucher list items.
- **Validating vouchers only on frontend**: Must have backend validation too. Frontend disables save button (UX), backend returns 400 (security).

## Common Pitfalls

### Pitfall 1: MIME Type Allowlist Gap
**What goes wrong:** Backend `file.controller.ts` has `ALLOWED_MIME_TYPES` that includes images + PDF + Excel but NOT Word documents. `file.constants.ts` also lacks Word MIME types and extensions.
**Why it happens:** Original file upload was designed for fabric images and Excel imports only.
**How to avoid:** Add Word MIME types to both `file.constants.ts` and `file.controller.ts`:
- `application/msword` (.doc)
- `application/vnd.openxmlformats-officedocument.wordprocessingml.document` (.docx)
- Also add `.doc` and `.docx` to `ALLOWED_EXTENSIONS`
**Warning signs:** Users unable to upload Word documents, getting "Invalid file type" errors.

### Pitfall 2: Transaction Atomicity for Payment + Vouchers
**What goes wrong:** Payment status updates and voucher record creation happen in separate DB operations. If one fails, data becomes inconsistent.
**Why it happens:** Easy to forget wrapping multi-table operations in a transaction.
**How to avoid:** Use `prisma.$transaction()` to wrap: (1) create PaymentRecord, (2) create PaymentVoucher entries, (3) update Order/SupplierPayment status -- all in one transaction.
**Warning signs:** Orphaned PaymentRecords without vouchers, or voucher records pointing to non-existent payment records.

### Pitfall 3: File Upload Timing vs Save
**What goes wrong:** Files are uploaded immediately on selection, but if user cancels the modal, orphaned files remain in storage.
**Why it happens:** The upload-immediately pattern means files exist before the payment form is submitted.
**How to avoid:** Accept this as a minor tradeoff -- orphaned files can be cleaned up by a periodic job later. Do NOT delay upload to save time (would block the modal submit). The current `File` table tracks all uploads; a cleanup job can find files not referenced by any `PaymentVoucher`.
**Warning signs:** Storage accumulating unreferenced files over time.

### Pitfall 4: SupplierPayment Upsert Race with Vouchers
**What goes wrong:** `updateSupplierPayment` uses upsert -- if it's the first payment for a supplier, the SupplierPayment is created. The PaymentRecord needs a valid orderId + supplierId, which must exist before voucher association.
**Why it happens:** The upsert pattern means we might be creating the SupplierPayment row for the first time.
**How to avoid:** Inside the transaction: (1) upsert SupplierPayment, (2) create PaymentRecord (orderId + supplierId now guaranteed to exist), (3) create PaymentVouchers.
**Warning signs:** Foreign key constraint violations on PaymentRecord.supplierId.

### Pitfall 5: Frontend Type Mismatch After DTO Changes
**What goes wrong:** Adding `voucherFileIds` to backend DTOs but forgetting to update frontend `UpdateCustomerPaymentData`/`UpdateSupplierPaymentData` types, or vice versa.
**Why it happens:** Backend and frontend types are manually synchronized.
**How to avoid:** Update both `forms.types.ts` (frontend) and DTO files (backend) in the same task.
**Warning signs:** TypeScript compilation errors, runtime 400 errors from unexpected fields.

### Pitfall 6: Ant Design Upload.Dragger File Type Validation
**What goes wrong:** `accept` prop alone doesn't reliably filter files on all browsers. Users might drag files of unsupported types.
**Why it happens:** Browser file input `accept` attribute is a hint, not enforcement.
**How to avoid:** Implement `beforeUpload` callback that checks both file extension and MIME type. Show error message for rejected files. Backend validates as final enforcement.
**Warning signs:** Unsupported file types reaching the backend and getting 400 errors.

### Backend: Payment Service with Voucher Creation
```typescript
// Source: Pattern from existing order-payment.service.ts transaction
async updateCustomerPayment(
  orderId: number,
  dto: UpdateCustomerPaymentDto,
  operatorId?: number,
): Promise<Order> {
  return this.prisma.$transaction(async (tx) => {
    await validateOrderExists(tx, orderId);

    // 1. Create PaymentRecord for audit trail
    const paymentRecord = await tx.paymentRecord.create({
      data: {
        orderId,
        type: 'customer',
        amount: dto.customerPaid ?? 0,
        payMethod: dto.customerPayMethod,
        remark: dto.notes,
        operatorId,
      },
    });

    // 2. Create PaymentVoucher entries
    const voucherData = dto.voucherFileIds.map((fileId, index) => ({
      paymentRecordId: paymentRecord.id,
      fileId,
      remark: dto.voucherRemarks?.[index] ?? null,
    }));
    await tx.paymentVoucher.createMany({ data: voucherData });

    // 3. Update order payment status
    const updateData = this.buildCustomerPaymentUpdateData(dto);
    return tx.order.update({
      where: { id: orderId },
      data: updateData,
      include: ORDER_INCLUDE_PAYMENT,
    });
  });
}
```

## Open Questions

1. **PaymentRecord: Should customer payment also go through PaymentRecord?**
   - What we know: Currently `updateCustomerPayment` directly updates Order fields. SupplierPayment has its own table.
   - What's unclear: Customer payment has no separate record per operation -- it just overwrites `customerPaid` on Order.
   - Recommendation: YES -- create a PaymentRecord for every payment update (both customer and supplier). This provides consistent audit trail and voucher association for both types. The `PaymentRecord.type = 'customer'` already supports this.

2. **File cleanup for orphaned uploads**
   - What we know: Files uploaded immediately, but user might cancel modal.
   - What's unclear: How many orphaned files will accumulate.
   - Recommendation: Accept orphans for MVP. Add a scheduled cleanup job in a future phase if needed. Track by checking `File` records not referenced in any `PaymentVoucher` or `FabricImage`.

3. **Existing voucher display for previous payments**
   - What we know: PaymentRecords will accumulate over time (multiple partial payments).
   - What's unclear: How to display historical vouchers vs. current-operation vouchers.
   - Recommendation: Display all vouchers grouped by PaymentRecord (chronological). The VoucherList shows all historical vouchers; the VoucherUploader only appears in the edit modal for the current operation.
