## Human Verification Required

### 1. Customer Payment Modal — Upload + Save Flow

**Test:** Start app, open any order, go to Payment tab, click Edit on customer payment card.
**Expected:** Upload area is visible below form fields with "Click or drag files to upload vouchers" text. Save button is disabled (greyed). Upload any image or PDF file. File appears in uploader list. Save button becomes enabled. Fill payment amount and status, click Save. Payment saves successfully and modal closes. Voucher appears in VoucherList below the payment card.
**Why human:** Modal open/close state, Ant Design button disabled attribute with real user interaction, and visual layout confirmation require a real browser.

### 2. Supplier Payment Modal — Upload + Save Flow

**Test:** Click Edit on any supplier payment card.
**Expected:** Same behavior as customer modal — upload area visible, save disabled without voucher, enabled after upload, payment saves with voucher.
**Why human:** Same reasons as above; supplier-specific modal behavior and supplierId filtering.

### 3. VoucherList File Type Actions

**Test:** After uploading different file types (image, PDF, Office document), click the action button for each in VoucherList.
**Expected:** Image voucher — "Preview" button opens an image preview modal in-page. PDF voucher — "Open" button opens the PDF in a new browser tab. Word/Excel voucher — "Download" button triggers browser file download.
**Why human:** Preview modal render, window.open behavior, and download link triggering cannot be automated in jsdom environment.

### 4. Customer vs Supplier Voucher Separation

**Test:** Upload a voucher for customer payment and a different voucher for a specific supplier payment. Navigate the payment tab.
**Expected:** Customer payment tab shows only the customer's voucher in VoucherList. Each supplier payment card shows only that supplier's vouchers below it.
**Why human:** Multi-supplier filtering and tab navigation require end-to-end verification with real data.
