## 一、项目概述

### 1.1 项目背景

铂润（Borealis Fabrics）是一家传统面料贸易公司，主营业务是从面料原厂采购面料，销售给大型家具公司。公司目前拥有约100家供应商，约50家B端客户，月均处理500+订单。

随着业务规模扩大，传统的人工处理方式已经难以支撑高效运营，公司决定进行数字化转型，建设信息化管理系统。

### 1.2 项目目标

通过数字化系统建设，实现以下目标：

1. **告别信息孤岛**：将分散在Excel、微信、纸质单据中的数据统一管理
2. **提升工作效率**：减少人工传话、重复沟通，实现信息自动同步
3. **规范业务流程**：建立标准化的询价、下单、跟踪流程
4. **智能辅助决策**：通过AI帮助快速查找面料、自动处理常规事务（远期目标）

### 1.3 系统范围

本项目计划建设两套核心系统：

| 系统名称 | 主要用途 | 优先级 | 使用者 | 平台 |
|---------|---------|--------|--------|------|
| **供应链自动化系统** | 管理供应商、采购订单、订单跟踪 | ⭐ 高（MVP一期） | 公司员工 | Web 工作台（桌面端为主）+ 企业微信 |
| **B2B电商平台** | 面向客户的自助平台，客户可浏览面料、下单、查询订单 | 中（MVP二期） | B端客户 | 微信小程序 |

### 1.4 技术架构（已确定）

| 项目 | 方案 | 说明 |
|------|------|------|
| 云服务 | 腾讯云 | 与微信生态集成更便捷 |
| 内部协作 | 企业微信 | 为后续微信自动化铺路 |
| 员工端 | Web 工作台 | 桌面端为主（99%场景），移动端响应式兼顾 |
| 客户端 | 微信小程序 | MVP二期开发 |

> **MVP阶段**：企业微信用于 OAuth 员工登录，Web 工作台以桌面浏览器为主要使用场景，移动端通过响应式设计兼顾。消息通知、审批流程等功能后期迭代

### 1.5 扩展性要求

- 未来公司考虑往跨境电商方向发展，系统需要有一定的扩展性和灵活性
- 预留AI智能功能接口
- 预留物流API对接接口
- 预留消息通知接口（后期接入腾讯云消息服务）

## 二、业务模式说明

### 2.1 商业模式

铂润采用**纯贸易中间商模式**：

```
┌──────────┐      ┌──────────┐      ┌──────────┐
│  面料供应商  │ ───→ │   铂润    │ ───→ │  家具公司客户 │
│ （约100家）  │      │ （信息撮合） │      │ （约50家B端）  │
└──────────┘      └──────────┘      └──────────┘
                        │
            ┌───────────┼───────────┐
            │           │           │
        订单管理    信息匹配     财务结算
```

**关键特点**：
- ✅ 采购模式：**按需采购**（客户下单后再向供应商采购，不提前备货）
- ❌ 铂润**没有自己的仓库**（但存在少量库存概念，见2.3）
- ❌ 铂润**不持有大量库存**
- ✅ 供应商**直接发货**给客户
- ✅ 铂润的价值：**信息撮合 + 订单管理 + 资金周转**
- ✅ 利润来源：**买卖价差**（供应商价格与客户价格之差，无固定比例，随市场定价）

### 2.2 付款模式

#### 2.2.1 客户 → 铂润

**主要模式（约90%业务）**：客户先付款给铂润

**其他模式（约10%业务）**：根据客户情况，支持多种方式：
- 预付
- 货到付款
- 账期（月结30天、月结60天等）

#### 2.2.2 铂润 → 供应商

**主要模式**：铂润收到客户付款后，再付款给供应商下单

**账期模式**：部分供应商允许铂润先生产、后付款（如账期90天）

> **重要说明**：账期是**双向**的——客户对铂润可能有账期，铂润对供应商也可能有账期。系统需要分别追踪两侧的付款状态。

### 2.3 库存概念（补充说明）

虽然铂润主要是贸易中间商，但存在少量库存：

| 库存类型 | 说明 |
|----------|------|
| 已付款库存 | 公司已经付款购入的库存 |
| 未付款库存 | 供应商寄售或预留的货物 |

> **MVP阶段**：库存管理作为后期迭代功能，MVP暂不实现

## 三、当前痛点分析

### 3.1 信息孤岛严重

| 问题 | 现状描述 | 影响 |
|-----|---------|------|
| 数据分散 | 采购、销售数据分散在Excel、微信聊天记录、纸质单据中 | 查找困难，无法统计分析 |
| 面料信息无汇总 | 上百个面料SKU的信息全在员工脑子里和手机相册中 | 依赖个人记忆，新员工难上手 |
| 订单记录不完整 | 订单状态靠微信沟通，没有统一记录 | 状态不清，容易遗漏 |

### 3.2 效率低下

| 问题 | 现状描述 | 影响 |
|-----|---------|------|
| 询价报价全靠人工 | 客户询价 → 员工查找 → 问供应商 → 回复客户，全程手动 | 响应慢，容易出错 |
| 员工当传话筒 | 客户和供应商有任何变动，都要员工手动转达 | 耗时耗力，信息可能丢失 |
| 面料查找困难 | 客户描述面料后，员工靠记忆+翻手机相册+去仓库找 | 效率极低，客户体验差 |
| 供应商账单格式不统一 | 员工往工厂下的订单得到的账单格式大不相同 | 多家供应商格式都需要人工调整，浪费时间且容易出错 |

### 3.3 财务对接（已调整）

| 问题 | 现状描述 | 解决方案 |
|-----|---------|----------|
| 系统独立 | 使用MYOB做财务管理，但与业务数据完全分离 | **本系统不与MYOB对接**，财务和业务提效分开，本系统专注实时业务信息 |

## 四、核心业务流程

### 4.1 标准订单流程

以下是一个典型订单从询价到完成的完整流程（已根据实际业务确认）：

```
阶段一：询价报价
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[客户] ──→ 通过微信/邮件询价，描述需要的面料
                    ↓
[铂润员工] ──→ 查找匹配的面料（目前靠记忆+相册）
                    ↓
[铂润员工] ──→ 向供应商询价，确认能否供货
                    ↓
[供应商] ──→ 回复报价和交期
                    ↓
[铂润员工] ──→ 给客户报价（有一定有效期）


阶段二：下单付款
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[客户] ──→ 确认接受报价，下单
                    ↓
[客户] ──→ 付款给铂润 ⭐（90%业务在此时付款，或走账期）
                    ↓
[铂润员工] ──→ 向供应商正式下单（付款或走账期）
                    ↓
[供应商] ──→ 确认订单，确认交货期


阶段三：生产质检
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[供应商] ──→ 安排生产
                    ↓
[供应商] ──→ 质量检验确认是否OK


阶段四：发货交付
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[供应商] ──→ 发送货物，提供物流信息
                    ↓
[铂润员工] ──→ 记录物流信息（物流公司、联系人、电话、单号）
                    ↓
[客户] ──→ 到达客户指定仓库


阶段五：对账收款
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[铂润员工] ──→ 对账
                    ↓
[客户] ──→ 收款完成（如有账期，此时付款）
                    ↓
[铂润员工] ──→ 向供应商付款（如有账期）
                    ↓
订单结束
```

### 4.2 订单状态定义（9个状态）

| 状态编号 | 状态名称 | 包含内容 | 备注 |
|----------|----------|----------|------|
| 1 | 询价中 | 客户咨询、公司确认价格 | 可取消 |
| 2 | 待下单 | 客户确认接受报价 | 可取消 |
| 3 | 已下单 | 客户下单、公司下发采购订单 | 可取消 |
| 4 | 生产中 | 确认交货期、安排生产 | 可取消 |
| 5 | 质检中 | 质量检验 | 可取消 |
| 6 | 已发货 | 发送货物、物流信息 | 可取消 |
| 7 | 已签收 | 到达客户仓库 | 可取消 |
| 8 | 已完成 | 对账、收款完成 | 可取消 |
| 9 | 已取消 | 订单取消 | 可恢复 |

> **取消与恢复规则**：
> - 所有状态（1-8）都可以取消，取消后状态变为"已取消"
> - 已取消的订单可以恢复到取消前的状态
> - **取消分场景规则**：
>   - 询价中/待下单：可直接取消，无特殊要求
>   - 已下单/生产中/质检中：取消时必须填写取消原因（备注字段）
>   - 已发货/已签收：取消时必须填写取消原因，且标注需后续处理退货（退货功能后期实现）
>   - 已完成：取消时必须填写取消原因
> - **退货处理**：MVP阶段不实现退货流程，已发货后取消仅做状态标记和原因记录，退货实物处理由员工线下沟通

### 4.3 期望的数字化流程改进

| 环节 | 现状 | MVP改进 | 远期改进 |
|-----|-----|---------|----------|
| 面料查找 | 员工靠记忆+翻相册 | 系统数据库检索 | AI根据客户描述自动匹配推荐 |
| 供应商询价 | 微信/邮件手动沟通 | 手动沟通（系统记录） | 系统自动发送询价邮件 |
| 供应商账单 | 手动调整格式 | 预留上传入口 | AI模型格式处理，自动录入 |
| 信息同步 | 员工手动转达 | 系统统一记录 | 系统自动通知各方 |
| 订单跟踪 | 微信聊天记录 | 系统统一记录，9个状态节点 | 物流API实时查询 |
| 财务记账 | 手动录入MYOB | 系统独立运作 | 保持独立 |

| 材质/成分 | 可多选 | 涤纶、棉、混纺 | 否 |
| 成分比例 | | 80%涤纶+20%棉 | 否 |
| 颜色 | 主色调 | 灰色、米白、深蓝 | 否 |
| 克重 | g/m² | 200、280、350 | 否 |

#### 5.1.3 图片管理

- 每款面料可上传多张产品图片
- **至少3张图片**（正面、细节、实物效果）

#### 5.1.4 关联信息：面料-供应商关系

一款面料可关联多家供应商，每个关联记录以下信息：

| 字段 | 说明 |
|------|------|
| 供应商 | 关联的供应商 |
| 采购价 | 该供应商的采购单价（元/米） |
| 最小起订量 | 该供应商的起订量要求（米） |
| 交货周期 | 该供应商的交货时间（天） |

> **说明**：实际采购时由员工根据价格、库存、交期等因素人工选择供应商

#### 5.1.5 客户特殊定价

可在面料维度设置特定客户的特殊价格。

> **定价逻辑**：支持从**面料维度**和**客户维度**双向设置特殊价格。系统优先级：客户特殊定价 > 面料默认销售价

#### 5.1.6 数据初始化

- 需要专人负责将现有上百个SKU的面料信息整理录入系统
- 预计整理时间：半个月到一个月
- **支持Excel批量导入**面料基础信息

> **Excel 导入冲突策略（已确认）**：编号已存在的行**跳过不导入**，返回错误提示（如"面料编号 BF-2601-0001 已存在，已跳过"）。成功的行正常导入，失败的行汇总报告给用户。不覆盖、不更新已有数据。

### 5.4 订单管理

**目的**：统一管理所有订单，实现全流程可追溯

#### 5.4.1 订单数据结构

订单采用**主表+明细子表**结构，支持一个订单包含多款面料，每款面料独立追踪状态。

**订单主表**：

| 字段 | 说明 |
|------|------|
| 订单编号 | 系统自动生成，格式 `ORD-YYMM-NNNN` |
| 客户信息 | 关联客户 |
| 收货地址 | 从客户地址列表中选择或手动输入 |
| 订单聚合状态 | 取所有明细中进度最低的状态，系统自动计算（见下方规则说明） |
| 订单总价 | 所有明细小计之和 |
| 订单备注 | MVP阶段可用于临时记录客诉等信息 |
| 创建时间 | |
| 更新时间 | |

> **订单聚合状态规则**：
> - 订单主表的聚合状态 = 所有明细中进度最低的状态
> - 状态进度排序：询价中 < 待下单 < 已下单 < 生产中 < 质检中 < 已发货 < 已签收 < 已完成
> - 已取消的明细不参与聚合计算
> - 如果所有明细都已取消，订单聚合状态为"已取消"
> - 举例：订单有3个明细，状态分别为"生产中""已发货""已签收"，则订单聚合状态为"生产中"

**订单明细子表**（每款面料一条记录）：

| 字段 | 说明 |
|------|------|
| 面料信息 | 关联面料 |
| 数量 | 单位：米 |
| 单价 | 元/米（销售价） |
| 小计 | 数量 × 单价 |
| 状态 | 9个状态之一（独立追踪） |
| 供应商 | 选择哪家供应商采购 |
| 采购价 | 元/米（该供应商的采购价） |
| 交期要求 | |
| 物流信息 | 关联物流记录（见5.6） |

#### 5.4.1a 多供应商订单采购流程说明

一个订单可包含多款面料，每款面料可选择不同的供应商采购。流程如下：

1. **创建订单**：选择客户，添加多个面料明细
2. **选择供应商**：每个明细独立选择供应商（可从面料-供应商关联列表中选择），采购价自动填充
3. **独立状态追踪**：每个明细有独立的状态（如面料A已发货、面料B仍在生产中）
4. **独立付款追踪**：供应商侧付款按 (订单, 供应商) 维度汇总追踪
5. **独立物流追踪**：每个明细可关联不同的物流记录（支持分批发货）

> 举例：客户下单3款面料，分别从供应商A、B、C采购。系统为每个供应商生成独立的付款追踪记录，各面料状态和物流信息独立管理。

#### 5.4.2 付款信息追踪

| 字段 | 说明 |

| 付款方式 | 微信/支付宝/银行卡/账期 |
| 账期天数 | 如月结30天（仅账期时填写） |
| 付款时间 | |

**供应商侧（应付）**：

| 字段 | 说明 |
|------|------|
| 应付金额 | 采购总成本（所有明细采购价小计之和） |
| 已付金额 | 已支付给供应商的金额 |
| 供应商付款状态 | 待付款/部分付款/已付清 |

> **技术实现备注**：由于一个订单可能包含多款面料分别从不同供应商采购，供应商侧付款按 (订单ID, 供应商ID) 维度独立追踪。详见技术需求文档第六章数据库设计。

#### 5.4.2a 付款与订单状态流转关系

> **已确认决策（MVP阶段）**：付款和订单状态**不绑定**，独立追踪。
>
> - 付款状态和订单状态各自独立变更，互不阻塞
> - 系统在以下场景仅做**提醒**（不强制阻塞）：
>   - 订单状态推进到"已发货"但客户尚未付款 → 前端显示提醒
>   - 订单状态推进到"已完成"但仍有未付清款项 → 前端显示提醒
> - 员工可自行决定是否继续推进，系统不做强制限制
> - 后期迭代：可增加付款阻塞规则（如未付款不可发货）

#### 5.4.2b 付款流水记录

#### 5.5.2 报价转订单流程

报价可一键转为订单（单面料报价 → 单明细订单）：

1. 仅**有效(active)** 且**未过期**的报价可转订单
2. 转换时自动创建订单，包含一条明细（面料、数量、单价从报价继承）
3. 转换后报价状态自动变为 **converted**（已转订单），不可再次转换
4. 订单明细初始状态为"待下单"(PENDING)
5. 订单时间线自动记录"从报价 QT-xxx 转换"

#### 5.5.3 使用方式

- 系统生成报价内容
- 员工手动复制发送给客户
- 报价过期后状态自动变为"已过期"，需重新报价
- 重大事件时价格调整需酌情而定

## 九、数据安全与运维要求

### 9.1 数据安全重要性声明

**数据安全是本系统最重要的需求**。系统承载了公司核心业务数据，包括客户信息、供应商信息、订单记录、报价记录等，数据丢失将对业务造成严重影响。

### 9.2 数据备份要求

在符合最大经济效益的前提条件下，采用**行业最高标准**进行数据安全备份，具体方案由技术团队设计，需满足：

- 定期自动备份
- 备份数据异地存储
- 备份数据可恢复验证
- 备份保留周期合理

### 9.3 系统可用性要求

- 系统需具备高可用性
- 故障时可快速恢复
- 具体SLA指标由技术团队根据成本效益确定

### 9.4 后期维护要求

- 系统需具备良好的可维护性
- 预留扩展接口
- 文档完善，便于后续交接
