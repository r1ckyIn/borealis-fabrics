# Context 管理策略

**优化 token 使用，避免 context 溢出。**

---

## 主动 /clear 建议

```text
Claude 应在以下情况建议 /clear：
- 对话超过 50 轮
- 切换到完全不同的任务
- 完成一个完整功能模块后
- 用户问"为什么变慢了"
```

---

## 避免重复读取

```text
Claude 应该：
✅ 记住已读取的文件内容
✅ 只读取需要的文件部分（offset/limit）
✅ 使用 Task(Explore) 委派大范围搜索
❌ 不要重复读取同一文件
❌ 不要读取整个大文件只为找一行
```

---

## 大任务分解

```text
当任务预计超过 30 轮对话时：
1. 将任务分解为独立的 session
2. 每个 session 完成一个模块/功能
3. session 之间用 CLAUDE.md 传递关键信息
4. 使用 Task tool 并行处理独立子任务
```

## Task Tool 使用指南

### 使用 Task(Explore) 的场景

- 搜索代码库中的模式
- 理解陌生模块的结构
- 查找特定实现的位置
- 开放式代码探索

### 使用 Task(Plan) 的场景

- 设计新功能的实现方案
- 评估重构影响范围
- 规划多步骤任务
- 架构设计

## Context 优化技巧

### 文件读取优化

```text
优化前：Read 整个 1000 行文件
优化后：Read 文件的特定范围（offset + limit）

优化前：多次读取同一文件
优化后：一次读取，缓存内容
```

### 搜索优化

```text
优化前：自己执行多轮 Grep/Glob
优化后：使用 Task(Explore) 委派

优化前：全局搜索
优化后：指定目录范围搜索
```

### 响应优化

```text
优化前：冗长的解释
优化后：简洁明了的回复

优化前：重复之前说过的内容
优化后：引用之前的讨论
```

## Context 溢出预警信号

| 信号 | 说明 | 处理 |
|------|------|------|
| 响应变慢 | 处理时间明显增加 | 建议 /clear |
| 对话轮数 > 50 | 累积太多上下文 | 建议 /clear |
| 重复读取文件 | 之前读过的文件又读 | 优化读取策略 |
| 任务混乱 | 多个任务混在一起 | 分解任务 |

## Session 间信息传递

### 通过 CLAUDE.md 传递

```text
在 CLAUDE.md 中记录：
- 当前进度
- 关键决策
- 待解决问题
- 下一步计划
```

### 通过进度表格传递

```markdown
| 功能 | 状态 | 测试 | Build | Lint | 备注 |
|------|------|------|-------|------|------|
| 功能 A | ✅ | ✅ | ✅ | ✅ | 已完成 |
| 功能 B | 🔄 | ⏳ | ⏳ | ⏳ | 进行中 |
| 功能 C | ⏳ | ⏳ | ⏳ | ⏳ | 待开始 |
```

## 并行任务处理

### 使用多个 Task 并行

当多个子任务相互独立时，可以同时启动多个 Task agent：

```text
并行搜索多个模块
并行运行多个测试
并行检查多个文件
```

### 顺序任务处理

当任务有依赖关系时，必须顺序执行：

```text
先完成功能 A → 再开始功能 B（B 依赖 A）
先运行测试 → 根据结果修复 → 再运行测试
```
