# 严格功能开发迭代流程

**核心原则：一个功能一个功能开发，不批量实现。** 每个功能必须经历完整的迭代周期，不可跳过任何步骤。

---

## 单功能开发周期（8 步）

```
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 1：使用计划skills规划功能                                      │
│ ├── 定义功能边界和验收标准                                        │
│ ├── 确定要修改/创建的文件                                         │
│ ├── 列出边界情况和错误处理                                         |
│ ├── 根据git工作流创建功能git分支，进行并行开发检查清单                  │
│ └── 获得用户批准后才继续在分支上开发                                  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 2：编写测试（TDD - 先写测试）                                 │
│ ├── 编写单元测试（覆盖正常路径 + 边界情况 + 错误情况）              │
│ ├── 运行测试 → 确认测试失败（红色）                                │
│ └── 测试代码也要 lint 检查                                        │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 3：实现功能                                                 │
│ ├── 使用LSP辅助编写最少代码以通过测试                               │
│ ├── 运行测试 → 确认测试通过（绿色）                                │
│ ├── 如果测试失败 → 修复 → 重新运行 → 反复直到通过                   │
│ └── 不要一次性写完所有代码，增量实现                               │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 4：验证循环（每次代码变更后必须执行）                          │
│ ├── 4.1 构建验证：pnpm build                                     │
│ │   └── 如果失败 → 修复 → 重新构建 → 反复                         │
│ ├── 4.2 测试验证：pnpm test                                      │
│ │   └── 如果失败 → 修复 → 重新测试 → 反复                         │
│ ├── 4.3 Lint 验证：pnpm lint                                     │
│ │   └── 如果有警告 → 修复 → 重新 lint → 反复                      │
│ └── 4.4 类型检查（前端）：pnpm typecheck                          │
│     └── 如果失败 → 修复 → 重新检查 → 反复                         │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 5：使用code reviewer进行严格自我代码审查                        │
│ ├── 代码质量和可读性                                              │
│ ├── 安全漏洞检查（SQL 注入、XSS、硬编码密钥等）                     │
│ ├── 边界情况是否都处理了                                          │
│ ├── 错误处理是否完整                                              │
│ └── 是否符合项目编码规范                                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 6：提交到 feature 分支                                       │
│ ├── /commit（使用标准提交格式，提交到 feature 分支）               │
│ └── 提交消息：feat/fix/test: <具体描述>                           │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 7：通过 PR 合并到 main（⚠️ 绝不直接 push main）               │
│ ├── 7.1 push feature 分支到远程：git push origin feature/xxx     │
│ ├── 7.2 创建 PR：/commit-push-pr 或 gh pr create                 │
│ ├── 7.3 代码审查：/code-review                                   │
│ ├── 7.4 修复审查问题（如有）→ /commit → 重新审查                   │
│ ├── 7.5 审查通过后：在 GitHub 上合并 PR                           │
│ └── 7.6 清理分支：/clean_gone                                    │
│                                                                 │
│ ⚠️ 禁止操作：                                                    │
│ ├── ❌ git checkout main && git merge feature/xxx               │
│ ├── ❌ git push origin main（本地合并后直接 push）                │
│ └── ❌ 跳过 /code-review 直接合并                                 │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 8：更新文档（强制）                                          │
│ ├── 更新 CLAUDE.md 的"当前功能状态"表格                           │
│ ├── 更新 VIBE_CODER_GUIDE 的进度状态（模块完成时）                 │
│ └── 记录任何新的技术决策或学习                                     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
              ✓ 功能完成 → 进入下一个功能
```

---

## 失败处理规则

如果任何验证步骤失败：

1. **立即停止** - 不要继续下一步
2. **分析错误原因** - 阅读错误信息
3. **修复问题** - 做最小改动
4. **重新运行所有验证步骤** - 确保修复没有引入新问题
5. **全部通过后才能继续** - 不可跳过

## ⚠️ 并行开发注意事项

**当多个 Claude Code 终端并行开发不同功能时，必须使用不同的 feature 分支。**

### 问题场景

```
❌ 错误做法：
终端 A：feature/stage2-xxx 开发功能 2.2.8
终端 B：feature/stage2-xxx 开发功能 2.3.8（同一分支）
→ 可能导致：一个终端完成后合并，另一个终端的代码被意外带入 main，绕过 PR 流程
```

### 正确做法

```
✅ 正确做法：
终端 A：feature/customer-pricing 开发功能 2.2.8-2.2.10
终端 B：feature/fabric-images 开发功能 2.3.8
→ 每个分支独立创建 PR，独立 code review，独立合并
```

### 并行开发检查清单

在开始新功能前，确认：

1. **检查当前分支**：`git branch --show-current`
2. **确认没有其他终端在同一分支工作**
3. **如果有并行任务，创建独立的 feature 分支**：
   - 分支命名要能区分功能：`feature/<module>-<feature>`
   - 例：`feature/customer-pricing`、`feature/fabric-images`
4. **每个分支独立完成 8 步周期后，独立创建 PR**
