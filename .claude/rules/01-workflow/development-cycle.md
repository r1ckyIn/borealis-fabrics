## 通用规则

### Git 分支策略与并行开发

**当多个 Claude Code 终端并行开发不同功能时，必须使用不同的 feature 分支。**

```
❌ 错误做法：
终端 A：feature/stage3-xxx 开发功能 3.2.1
终端 B：feature/stage3-xxx 开发功能 3.3.1（同一分支）
→ 可能导致：一个终端完成后合并，另一个终端的代码被意外带入 main

✅ 正确做法：
终端 A：feature/order-module 开发功能 3.2.x
终端 B：feature/logistics-module 开发功能 3.3.x
→ 每个分支独立创建 PR，独立 code review，独立合并
```

**并行开发检查清单**（在开始新功能前确认）：

1. **检查当前分支**：`git branch --show-current`
2. **确认没有其他终端在同一分支工作**
3. **如果有并行任务，创建独立的 feature 分支**：
   - 分支命名要能区分功能：`feature/<module>-<feature>`
   - 例：`feature/quote-module`、`feature/order-items`、`feature/import-module`
4. **每个分支独立完成开发周期后，独立创建 PR**

---

### 失败处理规则

如果任何验证步骤失败：

1. **立即停止** - 不要继续下一步
2. **分析错误原因** - 阅读错误信息
3. **修复问题** - 做最小改动
4. **重新运行所有验证步骤** - 确保修复没有引入新问题
5. **全部通过后才能继续** - 不可跳过

### 通用 PR 流程（步骤 5-9）

**所有工作流变体在特定步骤完成后，都执行以下通用流程：**

```
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 5：使用 code-simplifier 优化代码（强制）                      │
│ ├── 运行 code-simplifier 插件简化和优化代码                       │
│ ├── 提升代码清晰度、一致性和可维护性                               │
│ ├── 消除冗余代码和不必要的复杂度                                   │
│ └── 确保代码风格统一                                              │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 6：使用 code reviewer 进行严格自我代码审查                    │
│ ├── 代码质量和可读性                                              │
│ ├── 安全漏洞检查（SQL 注入、XSS、硬编码密钥等）                     │
│ ├── 边界情况是否都处理了                                          │
│ ├── 错误处理是否完整                                              │
│ └── 是否符合项目编码规范                                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 7：提交到 feature 分支                                       │
│ ├── /commit（使用标准提交格式，提交到 feature 分支）               │
│ └── 提交消息：feat/fix/test: <具体描述>                           │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 8：通过 PR 合并到 main（⚠️ 绝不直接 push main）               │
│ ├── 8.1 push feature 分支到远程：git push origin feature/xxx     │
│ ├── 8.2 创建 PR：/commit-push-pr 或 gh pr create                 │
│ ├── 8.3 代码审查：/code-review                                   │
│ ├── 8.4 修复审查问题（如有）→ /commit → 重新审查                   │
│ ├── 8.5 审查通过后：在 GitHub 上合并 PR                           │
│ └── 8.6 清理分支：/clean_gone                                    │
│                                                                 │
│ ⚠️ 禁止操作：                                                    │
│ ├── ❌ git checkout main && git merge feature/xxx               │
│ ├── ❌ git push origin main（本地合并后直接 push）                │
│ └── ❌ 跳过 /code-review 直接合并                                 │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤 9：更新文档（强制）                                          │
│ ├── 更新 ARCHITECTURE.md 的"当前功能状态"表格                      │
│ └── 记录任何新的技术决策或学习                                     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
              ✓ 功能完成 → 进入下一个功能
```
