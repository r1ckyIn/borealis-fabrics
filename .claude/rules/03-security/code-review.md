# 代码审查要求

本文件定义了代码审查的标准和流程。

---

## 代码审查要求（强制）

完成**每个功能**后，执行严格的代码审查：

- 代码质量和可读性
- 最佳实践合规性
- 潜在 bug 或边界情况
- 性能考量
- 安全漏洞
- 错误处理完整性

---

## 使用 /code-review

### 基本流程

1. 创建 PR 后运行 `/code-review`
2. 审查发现（confidence >= 80 通常是真实问题）
3. 如果发现问题：
   - 一次修复一个问题
   - 使用 `/commit` 提交消息：`fix(review): <具体描述>`
   - 重复 `/code-review` 直到通过

### 自动化审查

`code-review` 插件使用 4 个并行代理进行自动化 PR 审查。

**审查维度：**
- 代码质量
- 安全性
- 性能
- 最佳实践

## 外部代码审查反馈工作流

**当收到来自外部来源（其他 AI、人工审查者等）的代码审查反馈时：**

### 处理流程

```text
处理审查反馈：
1. 首先阅读并理解所有审查评论
2. 创建需要修复的项目清单
3. 一次修复一个审查项
4. 修复每个项后，立即用描述性消息提交
5. 只有在提交了前一个修复后才进入下一个
6. 所有修复完成后，推送到远程仓库
```

### 提交消息格式

```text
格式：fix(review): <简要描述修复内容>
对更改的内容要具体

示例：
✓ fix(review): add input validation for user email
✓ fix(review): rename variable x to userCount for clarity
✓ fix(review): add error handling in api request
✓ fix(review): remove unused import statements
✗ fix(review): address review comments（太笼统）
```

### 跟踪审查更改

- 每个提交 = 修复一个审查项
- 使用 `git log` 查看审查修复的完整历史
- 提交消息作为更改内容的文档

## 自我代码审查清单

### 代码质量

- [ ] 代码逻辑清晰，易于理解
- [ ] 变量和函数命名有意义
- [ ] 代码结构合理，无过度嵌套
- [ ] 无重复代码（DRY 原则）

### 安全性

- [ ] 无硬编码密钥
- [ ] 无 SQL/命令注入漏洞
- [ ] 输入验证完整
- [ ] 错误消息不泄露敏感信息

### 边界情况

- [ ] 空值处理
- [ ] 边界值处理
- [ ] 异常情况处理
- [ ] 并发安全（如适用）

### 错误处理

- [ ] 所有错误都被捕获和处理
- [ ] 错误消息对调试有帮助
- [ ] 不吞没错误
- [ ] 适当的错误传播

### 性能

- [ ] 无 N+1 查询问题
- [ ] 无不必要的循环
- [ ] 资源正确释放
- [ ] 合理的缓存策略

## confidence 阈值

| Confidence | 说明 | 处理 |
|------------|------|------|
| >= 80 | 通常是真实问题 | 必须修复 |
| 50-79 | 可能是问题 | 评估后决定 |
| < 50 | 可能是误报 | 可以忽略 |
